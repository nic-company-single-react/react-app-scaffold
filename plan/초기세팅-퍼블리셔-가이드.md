# [가이드] 퍼블리셔 — 폐쇄망 SI 스타일/퍼블리싱

> 대상: 퍼블리셔 (마크업·스타일·디자인 토큰 반영 담당)
> 목적: 공통 개발자가 세팅한 기반 위에서 브랜드 톤앤매너·레이아웃 스타일을 반영하고, 개발자에게 핸드오프한다.
> 관련: 반입·설정·빌드 등 인프라는 공통 개발자 몫 → [초기세팅-프론트공통개발자-가이드.md](./초기세팅-프론트공통개발자-가이드.md) 참조.
> 전제: 스타일 시스템은 **Tailwind v4 + Style Dictionary 토큰 + shadcn 시맨틱 vars + CSS Module(co-location)** 조합. `tailwind.config.js` 없음(v4 방식).

---

## 0. 시작 전 이해할 3가지

1. **작업 공간은 `src/publishing/`** — Figma → React 1차 작업 공간(라우터 미등록). 완료 후 개발자가 `domains/`·`shared/`로 이동(핸드오프).
2. **스타일 우선순위는 `app.css`의 import 순서** — 아래일수록 우선. 퍼블리셔 수정 구간은 하단 theme/layout 두 곳으로 격리돼 있음.
3. **토큰은 JSON이 원본(SSOT)** — 색상/간격/타이포는 `design-tokens/*.json`을 고치고 `npm run build:tokens`로 생성. 생성된 `assets/styles/tokens/*.css`는 **직접 편집 금지**.

## 1. 퍼블리싱 기본 규칙 (`publishing/README.md` 준수)

- [ ] **Tailwind 유틸리티 + `cn()`만 사용** — publishing 단계에서 **CSS 파일 생성 금지**.
- [ ] **토큰 클래스로 색상·그림자 참조** — 하드코딩 색상 대신 `bg-brand-500`, `text-foreground`, `shadow-theme-md` 등 토큰 기반 클래스.
- [ ] **경로 alias `@/` 사용** — 상대경로 남발 금지.
- [ ] **다크모드는 `dark:` 유틸** — 예 `bg-white dark:bg-neutral-900`. (프로젝트는 `.dark` 클래스 기반, `@custom-variant dark (&:is(.dark *))`)
- [ ] 핸드오프 시 컴포넌트를 `domains/<업무>/` 또는 `shared/`로 이동(개발자 협업).

## 2. 브랜드 색상 테마 작업 (공통 개발자와 협업)

> 스위치 연결(app.css import, layouts/index.ts)은 공통 개발자가 담당. 퍼블리셔는 **색상 값**을 만든다.

- [ ] **프로젝트 테마 파일 생성** — `src/assets/styles/themes/theme-example-project.css`(override 템플릿, 주석 예시) 복사 → `theme-<project>.css`.
      현재 `theme-default.css`는 niccompany(Sky Blue `#499ed8`) 예시 → **신규 SI는 교체 대상**.
- [ ] 테마 파일 3블록 이해하고 채우기:
  1. `@theme inline` — shadcn CSS vars(`--sidebar`/`--card`/`--primary` …)를 Tailwind 유틸로 브리지.
  2. `@theme` — `brand-25 ~ brand-950` 스케일을 시그니처 색으로 리매핑.
  3. `:root`(라이트) / `.dark`(다크) — background/foreground/card/primary/border/sidebar/chart 시맨틱 값.
- [ ] **간격/타이포/그림자 등 토큰 조정 시** — `design-tokens/primitive/*.json` + `semantic/light.json`·`dark.json` 수정 → 공통 개발자가 `npm run build:tokens` 실행(또는 절차 공유받아 직접).

## 3. 레이아웃/컴포넌트 스타일 반영

- [ ] **예제 카탈로그를 레퍼런스로 사용** — DEV 모드 사이드바의 52개 예제 페이지(`domains/example/pages/ui-components/*Component.tsx`).
      각 페이지가 컴포넌트 사용법 + 소스 + Props 표를 제공 → 실제 사용 패턴 참고.
- [ ] **shadcn 컴포넌트 재스타일 방법** — `data-slot` 속성 선택자로 오버라이드.
      실전 예시: `domains/example/components/ui-components/accordion/TransactionDetailAccordion.module.css`
      (`[data-slot="accordion-trigger"]` 등 + 금융권 톤 + 다크모드 `:global(.dark)`). 상세 주석 포함.
- [ ] **CSS Module이 필요한 경우(핸드오프 이후)** — co-location 규칙(CLAUDE.md):
  - 파일명 = 소비 주체 이름, 위치 = 소비 주체 바로 옆.
  - 재사용 UI 조각 → `<Component>.module.css`(컴포넌트 옆) / 페이지 전용 → `<Page>.module.css`(페이지 옆) / 도메인 공용 → `domains/<d>/common/styles/`.
  - 다크모드는 `:global(.dark) .클래스`.
- [ ] **레이아웃 조정** — 헤더/사이드바 스타일은 `shared/layouts/default/components/`(AppHeader, AppSidebar 등). 구조 변경은 공통 개발자와 협의.

## 4. 확인 & 핸드오프

- [ ] **Storybook으로 토큰/컴포넌트 확인** — `npm run storybook`, `DesignTokens.stories`(토큰 클래스 참조표), `introduction.mdx`.
- [ ] **라이트/다크 양쪽 검수** — 테마 토글로 두 모드 모두 확인.
- [ ] **앵커 이동 주의** — HashRouter라 `<a href="#id">` 금지(라우트 해시 깨짐). `document.getElementById(id)?.scrollIntoView(...)` 사용.
- [ ] **핸드오프** — `publishing/` 산출물을 개발자와 함께 `domains/`·`shared/`로 이동하고, publishing 단계에서 미룬 CSS Module화(필요분)를 이때 적용.
