# 단위테스트 인프라 — 작업 정리

> 이 스캐폴드에 **Vitest 기반 단위테스트 환경을 처음 세팅한 기록**.
> 목표: "설치한 라이브러리(용도)"와 "그것을 쓰기 위해 건드린 설정 4곳"을 남겨서, 나중에 누가 봐도 왜 이렇게 됐는지 알 수 있게 한다.

---

## 0. 큰 그림 — 두 종류의 작업

단위테스트는 두 단계로 나뉜다.

| 구분 | 내용 | 상태 |
|---|---|---|
| ① 인프라 작업 | 테스트를 "돌릴 수 있는" 환경 세팅 (설치 + 설정) | **완료 — 다시 안 건드림** |
| ② 테스트 작성 | 개발자가 컴포넌트/훅/스토어별 테스트 코드 작성 | 앞으로 계속 (연습 영역) |

이 문서는 **①번(인프라)** 을 어떻게 했는지에 대한 기록이다.

---

## 1. 채택한 조합과 이유

**Vitest + @testing-library/react + @testing-library/user-event** — Vite 기반 React의 사실상 표준 조합.

- **Vitest**: Vite 설정(`vite.config.ts`의 alias·플러그인)을 **그대로 재사용**한다. 그래서 Jest처럼 설정을 따로 중복해서 맞출 필요가 없음.
- **Testing Library**: "구현 방식(state 이름 등)"이 아니라 **"사용자가 화면에서 보는 것"** 기준으로 테스트하는 철학 → 리팩터링에 잘 안 깨지는 테스트를 유도.

---

## 2. 라이브러리 (용도별)

### 2-1. 이번에 새로 설치 (4개) — `npm install -D`

| 라이브러리 | 역할 | 설명 |
|---|---|---|
| **jsdom** | 가상 브라우저 | Node.js엔 `document`/`window`가 없다. 이걸 가짜로 만들어 컴포넌트를 "그릴" 수 있게 함 |
| **@testing-library/react** | 렌더링 + 조회 | `render()`로 jsdom에 컴포넌트를 그리고, `screen.getByText()` 등으로 요소를 찾음 |
| **@testing-library/user-event** | 사용자 행동 | 실제 사용자처럼 클릭·타이핑을 시뮬레이션 |
| **@testing-library/jest-dom** | DOM 검증문 | `toBeInTheDocument()` 같은 DOM 전용 매처를 `expect()`에 추가 |

> `-D` = `--save-dev`. 개발/테스트 전용 의존성이라 배포 빌드엔 안 들어간다.

### 2-2. 이미 있던 것 (설치 안 함)

| 라이브러리 | 역할 |
|---|---|
| **vitest** | 테스트 실행기(러너). 테스트를 찾아 돌리고 `expect` 문법 제공 |
| **@vitejs/plugin-react** | 테스트 중 JSX/TSX 변환 |

이 둘은 **Storybook(`@storybook/addon-vitest`)이 딸려 설치**해둔 상태라 재설치 불필요.
→ 이 프로젝트는 남들보다 설치할 게 적었다(러너는 이미 있었음).

### 2-3. 브라우저 모드용 (이미 설치돼 있음)

jsdom(가짜 DOM)만으로 부족한 경우(실제 레이아웃·크기 계산·CSS가 중요한 컴포넌트)를 위해,
**진짜 Chromium을 띄워서 테스트하는 "브라우저 모드"** 도 함께 구성해뒀다.

| 라이브러리 | 역할 |
|---|---|
| **@vitest/browser-playwright** | Vitest 브라우저 모드의 Playwright 프로바이더. `provider: playwright()` 로 연결 |
| **playwright** | 실제 브라우저(Chromium 등)를 구동하는 엔진 |

> 둘 다 `devDependencies`에 이미 있었다. 브라우저 실행 파일(Chromium)은 `npx playwright install chromium` 으로 받는다.
> jsdom과 브라우저 모드의 차이는 **"컴포넌트가 어디서 그려져 검사받느냐"** 일 뿐, 결과는 둘 다 터미널 pass/fail로 동일하다.

## 3. 설정 작업 (건드린 4곳)

설치만으로는 안 돌아간다. **"어디서(환경) / 뭘 미리 불러오고 / 어떻게 실행하고 / 타입은 어떻게 알려줄지"** 를 연결하는 4곳을 수정했다.

### 3-1. `vite.config.ts` — 테스트 환경 (핵심)

Vitest는 Vite 설정을 재사용하므로 **별도 설정 파일 없이** 여기에 `test` 블록을 붙인다.
이 프로젝트는 **하나의 Vitest 안에서 환경이 다른 두 묶음(`projects`)** 을 정의한다.

| 프로젝트 | 환경 | 대상 파일 | 용도 |
|---|---|---|---|
| `unit` | jsdom | `*.test.{ts,tsx}` | 대부분의 유닛 테스트 (빠름) |
| `browser` | 실제 Chromium(Playwright) | `*.browser.test.{ts,tsx}` | 실제 렌더링이 꼭 필요한 것만 (정확함) |

**나누는 기준은 폴더가 아니라 파일 이름 규칙.** 브라우저가 꼭 필요한 컴포넌트만 파일명을
`*.browser.test.tsx` 로 지으면 자동으로 브라우저에서 돈다. jsdom → 브라우저 전환도 **파일명만 바꾸면 끝.**

```ts
/// <reference types="vitest/config" />   // TS에게 test 옵션의 타입을 알려주는 줄
import { playwright } from '@vitest/browser-playwright';
// ...
test: {
  projects: [
    {
      extends: true,                          // 상위 vite 설정(alias·플러그인)을 상속
      test: {
        name: 'unit',
        environment: 'jsdom',                 // jsdom(가짜 DOM) 위에서 실행
        globals: true,                        // test/expect/describe를 import 없이 전역 사용
        setupFiles: './src/test/setup.ts',    // 모든 테스트 전에 자동 실행할 파일
        include: ['src/**/*.test.{ts,tsx}'],
        // 브라우저 전용 파일과 스토리북은 제외
        exclude: ['**/node_modules/**', '**/*.stories.*', 'src/**/*.browser.test.{ts,tsx}'],
      },
    },
    {
      extends: true,
      test: {
        name: 'browser',
        globals: true,
        setupFiles: './src/test/setup.ts',
        include: ['src/**/*.browser.test.{ts,tsx}'],
        exclude: ['**/node_modules/**', '**/*.stories.*'],
        browser: {
          enabled: true,
          provider: playwright(),             // 설치한 @vitest/browser-playwright 연결
          headless: true,                     // 창 없이 백그라운드로 Chromium 실행
          instances: [{ browser: 'chromium' }],
          api: { host: '127.0.0.1', port: 5199 },  // ⚠ Windows 예약 포트 회피 (아래 3-1-1)
        },
      },
    },
  ],
},
```

- `projects` — **환경별로 테스트를 갈라 돌리는 스위치.** 하나는 jsdom, 하나는 실제 브라우저.
- `extends: true` — 각 프로젝트가 상위 Vite 설정(`resolve.alias`, 플러그인 등)을 그대로 물려받게 함.
- `include` / `exclude` — 파일 이름 규칙으로 어느 프로젝트가 어떤 파일을 맡을지 가른다.
  `unit`은 `*.browser.test.*` 를 반드시 제외해야 두 프로젝트가 같은 파일을 중복 실행하지 않는다.
- `browser.provider: playwright()` — **설치한 `@vitest/browser-playwright`를 실제로 켜는 스위치.**

#### 3-1-1. ⚠ Windows 예약 포트 함정 (`api.port` 를 고정한 이유)

브라우저 모드는 내부적으로 로컬 서버를 하나 띄우는데, **기본값이 하필 Windows 예약 포트 대역의
IPv6 포트(예: `::1:63315`)를 잡으려다 `EACCES: permission denied` 로 실패**하는 경우가 있다.

- 원인: Windows가 Hyper-V/WinNAT용으로 특정 포트 대역을 예약해 둔다. 확인:
  ```
  netsh interface ipv4 show excludedportrange protocol=tcp
  ```
  (이 PC 기준 63252~63351 등이 예약돼 있고, 실패 포트 63315가 그 안에 들어감)
- 해결: `browser.api: { host: '127.0.0.1', port: 5199 }` 로 **IPv4 + 예약 대역 바깥의 고정 포트**를 지정.
- 다른 PC에서 또 `EACCES`가 나면 위 명령으로 예약 대역을 확인하고 `port` 값만 안전한 값으로 바꾸면 된다.

### 3-2. `src/test/setup.ts` — 공통 준비 파일 (새로 생성)

```ts
import '@testing-library/jest-dom';
```

이 한 줄이 실행되면 `toBeInTheDocument()` 등 매처가 전역 `expect`에 등록된다.
**3-1의 `setupFiles`가 이 파일을 모든 테스트 직전에 자동으로 한 번 불러준다** → 각 테스트 파일에서 jest-dom을 import할 필요가 없음.

### 3-3. `package.json` — 실행 스크립트

```json
"test": "vitest",                       // watch 모드 (개발 중 켜두기)
"test:run": "vitest run",               // 한 번 실행 후 종료 (CI/커밋 전)
"coverage": "vitest run --coverage"     // 커버리지 리포트
"test:ui": "vitest --ui"                // 브라우저 대시보드(@vitest/ui)로 결과를 보며 실행 — 목록·통과/실패·필터 (개발/디버깅용)
```

### 3-4. `tsconfig.app.json` — 타입 인식

```json
"types": ["vite/client", "vitest/globals", "@testing-library/jest-dom"],
```

`globals: true`로 전역에서 `test`/`expect`를 쓰는데 TS는 그걸 모르니 빨간 줄이 뜬다. 그래서:
- `vitest/globals` — 전역 `test`/`expect`/`describe` 타입 인식
- `@testing-library/jest-dom` — `toBeInTheDocument` 등 매처 타입 인식

## 4. 동작 흐름 한눈에

```
npm test 실행
   ↓
vitest가 vite.config.ts의 test.projects 를 읽음
   ↓
파일 이름 규칙으로 두 묶음으로 갈라짐
   ├─ *.test.tsx          → [unit]    environment: jsdom (가짜 DOM)
   └─ *.browser.test.tsx  → [browser] 실제 Chromium(Playwright) 실행
   ↓
두 묶음 모두 setupFiles(src/test/setup.ts)로 jest-dom 매처 등록
   ↓
각 테스트: render(리액트) → 요소 찾기 → expect로 검증
   ↓
결과는 둘 다 터미널 pass/fail 로 동일하게 표시
```

**설치(jsdom 4개 + 브라우저 2개) → 환경 스위치(projects: jsdom/browser) → 공통 준비(setup) → 실행 명령(script) → 타입 연결(tsconfig).**
이 5가지가 인프라 작업의 전부.

## 5. 실행 명령어 (일상 사용)

```bash
npm test          # watch 모드 — 저장하면 관련 테스트만 자동 재실행 (개발 중 이거 켜두기)
npm run test:run  # 한 번만 실행하고 종료 (CI/커밋 전 확인용)
npm run coverage  # 어느 코드가 테스트됐는지 커버리지 리포트
npm run test:ui   # 브라우저 대시보드(@vitest/ui)로 결과를 보며 실행 — 목록·통과/실패·필터 (개발/디버깅용)

# 특정 묶음만 골라 실행 (프로젝트 이름으로)
npx vitest --project unit      # jsdom 만 (평소 개발 중 빠르게)
npx vitest --project browser   # 브라우저(Chromium) 만
```

> 참고: 위 `npm test`/`test:run` 은 **두 묶음(unit + browser)을 모두** 돈다.
> 평소엔 `--project unit` 로 빠르게 돌리고, 커밋 전이나 CI에서만 전체를 돌리는 운영도 가능.

## 6. 테스트 작성 규칙 (개발자 영역)

### 6-1. 파일 규칙
- **기본(jsdom):** 테스트할 파일 옆에 `xxx.test.tsx`(또는 `.test.ts`)로 만든다. → `unit` 프로젝트에서 실행.
- **브라우저가 꼭 필요할 때만:** `xxx.browser.test.tsx` 로 만든다. → `browser` 프로젝트(실제 Chromium)에서 실행.
  - 실제 레이아웃/크기 계산/CSS·hover 등 **jsdom으로는 부정확한 것**을 검증할 때만 쓴다.
  - jsdom 테스트를 브라우저로 올리고 싶으면 **파일명을 `.test.tsx` → `.browser.test.tsx` 로 바꾸기만** 하면 된다.
- **설정은 다시 건드릴 필요 없음.** 파일 이름 규칙만 지키면 Vitest가 알아서 맞는 묶음으로 보낸다.

### 6-2. 사고 흐름 (외울 3단계)
1. **Arrange(준비)** — `render(<컴포넌트 props />)` 로 화면에 그림
2. **Act(행동)** — 필요 시 `await userEvent.click(...)` / `.type(...)`
3. **Assert(검증)** — `expect(screen.getByRole/Text(...)).toBe...()`

### 6-3. 요소 찾기 우선순위 (Testing Library 권장)
- `getByRole('button', { name: '저장' })` ← 가장 권장(접근성 기준 = 실제 사용자 인식 방식)
- `getByText(...)`, `getByLabelText(...)`
- `get*`(없으면 에러) / `query*`(없으면 null, "없음" 검증용) / `find*`(비동기, 나중에 나타나는 것)
