# 단위테스트 인프라 — 작업 정리

> 이 스캐폴드에 **Vitest 기반 단위테스트 환경을 처음 세팅한 기록**.
> 목표: "설치한 라이브러리(용도)"와 "그것을 쓰기 위해 건드린 설정 4곳"을 남겨서, 나중에 누가 봐도 왜 이렇게 됐는지 알 수 있게 한다.

---

## 0. 큰 그림 — 두 종류의 작업

단위테스트는 두 단계로 나뉜다.

| 구분 | 내용 | 상태 |
|---|---|---|
| ① 인프라 작업 | 테스트를 "돌릴 수 있는" 환경 세팅 (설치 + 설정) | **완료 — 다시 안 건드림** |
| ② 테스트 작성 | 개발자가 컴포넌트/훅/스토어별 테스트 코드 작성 | 앞으로 계속 (연습 영역) |

이 문서는 **①번(인프라)** 을 어떻게 했는지에 대한 기록이다.

---

## 1. 채택한 조합과 이유

**Vitest + @testing-library/react + @testing-library/user-event** — Vite 기반 React의 사실상 표준 조합.

- **Vitest**: Vite 설정(`vite.config.ts`의 alias·플러그인)을 **그대로 재사용**한다. 그래서 Jest처럼 설정을 따로 중복해서 맞출 필요가 없음.
- **Testing Library**: "구현 방식(state 이름 등)"이 아니라 **"사용자가 화면에서 보는 것"** 기준으로 테스트하는 철학 → 리팩터링에 잘 안 깨지는 테스트를 유도.

---

## 2. 라이브러리 (용도별)

### 2-1. 이번에 새로 설치 (4개) — `npm install -D`

| 라이브러리 | 역할 | 설명 |
|---|---|---|
| **jsdom** | 가상 브라우저 | Node.js엔 `document`/`window`가 없다. 이걸 가짜로 만들어 컴포넌트를 "그릴" 수 있게 함 |
| **@testing-library/react** | 렌더링 + 조회 | `render()`로 jsdom에 컴포넌트를 그리고, `screen.getByText()` 등으로 요소를 찾음 |
| **@testing-library/user-event** | 사용자 행동 | 실제 사용자처럼 클릭·타이핑을 시뮬레이션 |
| **@testing-library/jest-dom** | DOM 검증문 | `toBeInTheDocument()` 같은 DOM 전용 매처를 `expect()`에 추가 |

> `-D` = `--save-dev`. 개발/테스트 전용 의존성이라 배포 빌드엔 안 들어간다.

### 2-2. 이미 있던 것 (설치 안 함)

| 라이브러리 | 역할 |
|---|---|
| **vitest** | 테스트 실행기(러너). 테스트를 찾아 돌리고 `expect` 문법 제공 |
| **@vitejs/plugin-react** | 테스트 중 JSX/TSX 변환 |

이 둘은 **Storybook(`@storybook/addon-vitest`)이 딸려 설치**해둔 상태라 재설치 불필요.
→ 이 프로젝트는 남들보다 설치할 게 적었다(러너는 이미 있었음).

## 3. 설정 작업 (건드린 4곳)

설치만으로는 안 돌아간다. **"어디서(환경) / 뭘 미리 불러오고 / 어떻게 실행하고 / 타입은 어떻게 알려줄지"** 를 연결하는 4곳을 수정했다.

### 3-1. `vite.config.ts` — 테스트 환경 (핵심)

Vitest는 Vite 설정을 재사용하므로 **별도 설정 파일 없이** 여기에 `test` 블록을 붙인다.

```ts
/// <reference types="vitest/config" />   // TS에게 test 옵션의 타입을 알려주는 줄
// ...
test: {
  environment: 'jsdom',                    // 테스트를 jsdom(가상 브라우저) 위에서 실행
  globals: true,                           // test/expect/describe를 import 없이 전역 사용
  setupFiles: './src/test/setup.ts',       // 모든 테스트 전에 자동 실행할 파일
  exclude: ['**/node_modules/**', '**/*.stories.*'],  // 스토리북 파일은 대상 제외
},
```

- `environment: 'jsdom'` — **설치한 jsdom을 실제로 켜는 스위치.** 없으면 `document 없음` 에러.
- `globals: true` — 파일마다 `import { test, expect } from 'vitest'` 안 써도 되게 함(Jest 스타일).
- `setupFiles` — 아래 3-2를 가리킴.
- `exclude` — Storybook `*.stories.tsx`를 단위테스트로 오해하지 않도록 제외.

### 3-2. `src/test/setup.ts` — 공통 준비 파일 (새로 생성)

```ts
import '@testing-library/jest-dom';
```

이 한 줄이 실행되면 `toBeInTheDocument()` 등 매처가 전역 `expect`에 등록된다.
**3-1의 `setupFiles`가 이 파일을 모든 테스트 직전에 자동으로 한 번 불러준다** → 각 테스트 파일에서 jest-dom을 import할 필요가 없음.

### 3-3. `package.json` — 실행 스크립트

```json
"test": "vitest",                       // watch 모드 (개발 중 켜두기)
"test:run": "vitest run",               // 한 번 실행 후 종료 (CI/커밋 전)
"coverage": "vitest run --coverage"     // 커버리지 리포트
```

### 3-4. `tsconfig.app.json` — 타입 인식

```json
"types": ["vite/client", "vitest/globals", "@testing-library/jest-dom"],
```

`globals: true`로 전역에서 `test`/`expect`를 쓰는데 TS는 그걸 모르니 빨간 줄이 뜬다. 그래서:
- `vitest/globals` — 전역 `test`/`expect`/`describe` 타입 인식
- `@testing-library/jest-dom` — `toBeInTheDocument` 등 매처 타입 인식

## 4. 동작 흐름 한눈에

```
npm test 실행
   ↓
vitest가 vite.config.ts의 test 블록을 읽음
   ↓
environment: jsdom  → 가상 브라우저 켬
setupFiles          → src/test/setup.ts 실행 → jest-dom 매처 등록
   ↓
*.test.tsx 파일들을 찾아서 실행
   ↓
각 테스트: render(리액트) → screen으로 찾기 → expect로 검증
```

**설치(4개) → 환경 스위치(jsdom) → 공통 준비(setup) → 실행 명령(script) → 타입 연결(tsconfig).**
이 5가지가 인프라 작업의 전부.

---

## 5. 실행 명령어 (일상 사용)

```bash
npm test          # watch 모드 — 저장하면 관련 테스트만 자동 재실행 (개발 중 이거 켜두기)
npm run test:run  # 한 번만 실행하고 종료 (CI/커밋 전 확인용)
npm run coverage  # 어느 코드가 테스트됐는지 커버리지 리포트
```

## 6. 테스트 작성 규칙 (개발자 영역)

### 6-1. 파일 규칙
- 테스트할 파일 옆에 `xxx.test.tsx`(또는 `.test.ts`)로 만든다. 이름에 `.test.`만 있으면 Vitest가 자동으로 찾는다.
- **설정은 다시 건드릴 필요 없음.** 파일만 만들면 됨.

### 6-2. 사고 흐름 (외울 3단계)
1. **Arrange(준비)** — `render(<컴포넌트 props />)` 로 화면에 그림
2. **Act(행동)** — 필요 시 `await userEvent.click(...)` / `.type(...)`
3. **Assert(검증)** — `expect(screen.getByRole/Text(...)).toBe...()`

### 6-3. 요소 찾기 우선순위 (Testing Library 권장)
- `getByRole('button', { name: '저장' })` ← 가장 권장(접근성 기준 = 실제 사용자 인식 방식)
- `getByText(...)`, `getByLabelText(...)`
- `get*`(없으면 에러) / `query*`(없으면 null, "없음" 검증용) / `find*`(비동기, 나중에 나타나는 것)
