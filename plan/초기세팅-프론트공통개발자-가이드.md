# [가이드] 프론트엔드 공통 개발자 — 폐쇄망 SI 초기 세팅

> 대상: 프론트엔드 공통 개발자 (기반 세팅 담당)
> 목적: 업무 개발자 투입 즉시 개발 가능하도록 반입~기반작업을 완성한다.
> 전제: 의존성은 **node_modules 통째 반입** / 배포 방식 **미정(서브경로·루트 양쪽 준비)** / HashRouter 고정.
> 관련: 퍼블리셔 작업은 [초기세팅-퍼블리셔-가이드.md](./초기세팅-퍼블리셔-가이드.md) 참조. 공통 개발자는 퍼블리셔에게 테마·규칙을 **가이드로 제공**한다.

## Phase 0. 반입 전 사전 준비 (외부망/인터넷 환경)

- [ ] **Node 버전 확정·고정** — 현재 `engines`/`.nvmrc` 없음. 의존성이 vite8/ts6/@types/node24 계열이라 **Node 20+ (권장 22·24)**.
      → `.nvmrc` 추가 또는 반입 문서에 버전 명시 + **Node 설치본 함께 반입**.
- [ ] **`npm ci`로 재현 설치**(package-lock 기준) → `node_modules/` 통째 압축. `npm install`(lock 갱신)이 아닌 `ci` 사용.
- [ ] **Playwright 브라우저 바이너리 결정** — `@vitest/browser-playwright`가 Chromium 별도 다운로드 필요.
      폐쇄망에서 `*.browser.test.tsx`(vitest `browser` project) 미동작 → (a) 브라우저 캐시(`~/.cache/ms-playwright`) 반입, or (b) 브라우저 테스트 제외·jsdom `unit`만 운영. **jsdom 단위테스트는 정상.**
- [ ] **빌드 사전 검증** — `npm run build` (= `build:tokens` → `tsc -b` → `vite build`) 성공 확인 후 반입.
- [ ] **외부 의존 문서 스냅샷** — README 가이드 링크(`redsky02122.dothome.co.kr`)·Storybook은 폐쇄망 접근 불가 → 캡처/PDF 또는 `build-storybook` 정적본 반입.
- [ ] **반입 목록 확정** — 소스(권장 `git bundle`, or zip) + `node_modules.zip` + Node 설치본 + (선택)브라우저 캐시 + 문서 스냅샷.
      폰트(`@fontsource-variable/geist`, `assets/fonts/outfit`)는 이미 번들 → CDN 의존 없음.

## Phase 3. 환경/설정 커스터마이징 (핵심 수정 지점)

수정 지점은 **`.env` 계열 + `src/config/*.config.ts` 뿐**(core 불가침). `.gitignore`가 `*.local`만 제외 → 사이트별 비밀값은 `.env.local`.

- [ ] **3환경(로컬/개발/운영) 값 정의**:
  - `VITE_API_BASE_URL` — 개발 `.env`가 외부 도메인(`jsonplaceholder.typicode.com`) → **폐쇄망 내부 API 주소/상대경로(`/`)로 교체(필수)**.
  - `VITE_SERVER_URL` — `vite.config.ts` dev proxy `/api` 타겟 = 내부 API 서버.
  - `VITE_BASE_URL` — 배포 서브경로(**끝 슬래시 필수**: index.html `%VITE_BASE_URL%logo.ico`). 배포 미정 → **서브경로(`/axiom/react/`)·루트(`/`) 양쪽 값 준비**.
  - `VITE_ROUTER_BASENAME` — HashRouter라 `/` 유지.
  - `VITE_LOCALSTORAGE_TOKEN_NAME` / `VITE_THEME_STORAGE_KEY` / `VITE_SPLASH_*`.
- [ ] **`src/config/` 조정** — `api.config.ts` / `query.config.ts` / `auth.config.ts` / `router.config.ts` / `theme.config.ts`. **SI가 만지는 유일한 설정 레이어.**
- [ ] **API 연동 전략 확정**(배포 미정 → 양쪽 문서화):
  - 동일 origin: `VITE_API_BASE_URL=/` (상대경로 → `api-client.ts`가 `window.location.origin` 기준 절대화). HashRouter라 **서버 rewrite 불필요**.
  - 별도 backend origin: 절대 URL + CORS/프록시 고려.
- [ ] **브랜드 자산 교체** — `index.html`, `logo.ico`, favicon, 스플래시 색상.

## Phase 4. 레이아웃·테마 기반작업 (공통 개발자 몫 + 퍼블리셔 협업)

> 색상/톤앤매너 값 자체는 퍼블리셔가 [초기세팅-퍼블리셔-가이드.md](./초기세팅-퍼블리셔-가이드.md)에서 작업. 공통 개발자는 **연결·스위치·빌드·구조**를 담당.

- [ ] **테마/레이아웃 스위치 교체(딱 4곳)** — `app.css`의 theme import + layout import 교체, `shared/layouts/index.ts`의 `RootLayout` export.
      레이아웃 커스터마이즈 시 `shared/layouts/default/` 폴더 통째 복제 후 스위치만 교체(자체완결형 모듈).
- [ ] **디자인 토큰 파이프라인** — `design-tokens/*.json`(SSOT) 수정 → `npm run build:tokens` → `assets/styles/tokens/*.css` **자동생성(직접 편집 금지)**.
- [ ] **네비게이션 재구성** — `shared/layouts/default/config/navigation.tsx`를 실제 업무 메뉴로 교체. `AppHeader`의 Github 버튼 등 하드코딩 URL 제거/교체.
- [ ] **다크모드 정책 확정** — 자체 `ThemeProvider`(`.dark` 클래스 + `VITE_THEME_STORAGE_KEY`). 퍼블리셔에게 CSS Module 규칙(`:global(.dark) .클래스`) 전달.
- [ ] **퍼블리셔에게 가이드 전달** — [초기세팅-퍼블리셔-가이드.md](./초기세팅-퍼블리셔-가이드.md) + 예제 카탈로그(DEV 52개 페이지) + 토큰 참조표(Storybook `DesignTokens.stories`) 안내.

## Phase 5. 프로젝트 구조·컨벤션 확정 (업무 개발자 투입 대비)

- [ ] **example/publishing 정리 정책** — 둘 다 `import.meta.env.DEV`에서만 라우터 등록(운영 자동 제외). → 학습 레퍼런스로 유지 권장(운영 번들 영향 없음).
- [ ] **업무 도메인 뼈대 규칙** — `domains/<업무>/` 하위 `pages/`·`components/`(pages 미러링)·`store/`·`router/`. CLAUDE.md 폴더 규칙 준수.
- [ ] **라우터 등록 3스텝 문서화** — 페이지 생성 → `domains/<d>/router/index.tsx`에 `loadable(() => import(...))` + `{path, element, name}` → `navigation.tsx` 메뉴 추가.
- [ ] **상태관리 배치 규칙** — `defineStore`/`createStore`(`@axiom/store`)만 사용. 도메인전용 `domains/<d>/store/`, 공유 승격 `shared/store/`, core 전역 `core/ui/store.ts`.
- [ ] **공통 인프라 사용법 온보딩** — `$util`/`$ui`/`$router` 전역, `@axiom/components/ui` 배럴(shadcn 37종), `SmartTable`(서버사이드 정렬·페이징·CSV/XLSX), react-query, `api-client`.
- [ ] **코드 컨벤션 고정** — ESLint(flat, semi, jsx double-quote) + Prettier(**탭, printWidth 120, singleQuote**) + `.vscode/settings.json`(formatOnSave). 커밋/브랜치 전략 합의.

## Phase 6. 배포·검증 & 개발자 온보딩

- [ ] **배포 인프라 준비** — 리포에 Dockerfile/nginx/CI 없음 → 정적 서빙 웹서버 설정 별도 마련. **HashRouter라 SPA rewrite 불필요**(dist 정적 배치만으로 동작).
- [ ] **정적서빙 검증**(배포 미정 → 양쪽) — 서브경로(`VITE_BASE_URL=/path/`) 빌드 + 루트(`/`) 빌드 각각 `npm run preview`/실제 웹서버 스모크 테스트(라우팅·자산·API).
- [ ] **신규 개발자 세팅 절차 시연** — 소스 clone + node_modules 배포본 전개 → `npm run dev` 확인(Phase 2 배포 방안 검증).
- [ ] **샘플 업무 도메인 1개** — 라우팅·API·SmartTable·폼 end-to-end 시연(온보딩 레퍼런스).
