Vitest 단위테스트 인프라 배선도

npm test 한 번이 어떤 경로로 흘러 컴포넌트를 렌더링하고 검증까지 도달하는지 — 그리고 그 흐름을 만들기 위해 건드린 설정 4곳이 서로 어떻게 연결되는지를 한 장으로 정리했습니다.

도식 1 연결 구조 — 설정 4곳의 배선

초록 선은 자동으로 흐르는 경로, 회색 점선은 보조 설정의 연결, 앰버 점선 박스는 아직 사람이 작성해야 하는 영역입니다.

초록 = 자동으로 흐르는 경로 / 이미 세팅 완료 앰버 점선 = 아직 사람이 작성할 영역 (테스트 파일)

도식 2 라이브러리 계층 — 누가 무슨 일을 하나

하나의 테스트 요청이 위 러너에서 아래 검증까지 층층이 협력합니다. jsdom 계열 4개와 브라우저 모드 2개를 -D로 설치했고, 맨 위 러너는 Storybook이 딸려 깔아둔 것을 재사용했습니다.

요약 핵심 4가지

1 · Vite 설정 재사용

별도 jest.config 없이 vite.config.ts 안에 test: {} 블록만 추가. alias @·@axiom/* 가 테스트에서도 그대로 통함.

2 · setupFiles = 자동 준비물

setup.ts에서 jest-dom 매처를 한 번만 등록 → 각 테스트 파일에서 매번 import 하지 않아도 됨.

3 · 인프라는 끝, 이제 파일만

설정은 다시 안 건드림. 파일 옆에 xxx.test.tsx 를 만들면 Vitest가 자동으로 찾아 실행.

4 · 브라우저가 필요하면

파일명을 xxx.browser.test.tsx 로 지으면 실제 Chromium(Playwright)에서 실행. Windows에선 예약 포트 회피용으로 browser.api.port를 고정해 둠.

명령 일상 사용

npm testwatch 모드 — 저장하면 관련 테스트만 자동 재실행 (개발 중)
npm run test:run1회 실행 후 종료 — CI / 커밋 전 확인용
npm run coverage어느 코드가 테스트됐는지 커버리지 리포트
npx vitest --project unitjsdom 묶음만 — 평소 개발 중 빠르게
npx vitest --project browser브라우저(Chromium) 묶음만
